# Product Handoff For New AI

이 문서는 기존 LinkLens를 **새 이름으로 다시 시작할 AI**를 위한 제품 브리프다.  
기술 선택은 AI가 자율적으로 하되, 아래 목적/기능/작업 방식은 반드시 지킨다.

## 세션 시작 루틴 (업무 연속성 필수)

매 작업 세션 시작 시 아래 파일을 **순서대로 반드시 읽는다**.

1. `README.md` (전체 원칙/목표)
2. `DECISIONS.md` (아키텍처/정책 결정 배경)
3. `KNOWN_ISSUES.md` (현재 재현 버그/우회책)
4. `NEXT_TASKS.md` (앞으로 할 일/우선순위)
5. `docs/SETUP_KO.md` (로컬/배포 연결 방법)
6. `HISTORY.md` (최근 변경 이력, 최신 섹션부터)

세션 종료 전에는 최소 아래를 반영한다.

- 완료한 작업: `HISTORY.md`에 기록
- 다음으로 이어야 할 작업: `NEXT_TASKS.md` 상태 갱신
- 새로 발견한 장애: `KNOWN_ISSUES.md` 갱신

## 이 웹페이지의 목적

- 사용자가 웹에서 본 기사/콘텐츠 링크를 빠르게 저장한다.
- 저장한 링크를 나중에 쉽게 다시 찾고, 정리하고, 읽는다.
- AI 보조로 제목/요약/키워드가 자동 보강되어 정리 부담을 줄인다.
- 개인별 아카이브처럼 쌓이되, 사용 흐름은 항상 가볍고 즉각적이어야 한다.

## 핵심 사용자 경험

- 링크를 넣으면 최소 입력으로 바로 저장된다.
- 저장 직후 목록에서 즉시 확인 가능해야 한다.
- 읽기 상태, 즐겨찾기, 컬렉션으로 정리할 수 있어야 한다.
- 검색/필터/정렬로 많은 링크에서도 빠르게 찾을 수 있어야 한다.
- AI 분석 실패가 있어도 저장/조회 흐름은 끊기지 않아야 한다.
- 모르는 정보가 있으면 추측, 할루시네이션 하지말아야함. 반드시 질문 후 작업을 진행하도록.

## 필수 기능

- 로그인, 회원가입, 로그아웃
- 링크 저장
- 링크 수정(메모, 상태, 별점, 태그)
- 삭제 및 복원(휴지통)
- 컬렉션 생성/수정/삭제 및 링크 할당
- 카테고리/컬렉션/상태 기반 필터
- 검색 및 정렬
- 카드/리스트 보기 전환
- AI 제목 미리보기, AI 요약/키워드/카테고리 분석

## 품질 목표

- 새로고침 후 데이터가 늦게 뜨는 체감이 없어야 한다.
- 로그인 상태 전환(로그인/로그아웃/세션복구) 중 화면이 꼬이지 않아야 한다.
- 네트워크가 느리거나 실패해도 사용자는 “멈춤”이 아닌 “복구 가능한 상태”를 보게 해야 한다.
- 배포 후에도 기능 불일치가 생기지 않도록 단일 기준 소스를 유지한다.

## 작업 우선순위

1. 저장/조회/로그인 같은 기본 흐름을 끊김 없이 완성
2. 새로고침/재접속/느린 네트워크 상황 안정화
3. 필터/검색/정리 UX 고도화
4. AI 품질 향상
5. 운영 편의성(로그/에러 추적) 강화

## 사용자 성향 (README 기반)

- 문제를 재현한 뒤 즉시 수정하고, 가능한 한 한 번에 마무리하는 작업 흐름을 선호함
- 설명은 길게 이론화하기보다 짧고 명확한 실행 결과를 선호함
- 한국어 커뮤니케이션을 선호함
- “사용자 흐름이 끊기지 않게” 안정성을 매우 중요하게 봄
- 배포/실서비스에서 실제로 동작하는지 확인하는 것을 중요하게 봄

## 사용자 협업 성향 (누적 반영)

- "왜 안됐는지" 분석보다 "지금 무엇을 고치고 어떻게 확인할지"를 먼저 원함
- 반영 누락 의심 시 `커밋 -> 푸시 -> 배포 -> 캐시` 순서의 즉시 확인을 선호함
- 요청 범위를 벗어난 확장 작업보다, 지시한 이슈를 우선 짧게 끝내는 흐름을 선호함
- 작업 완료 시 자동 커밋/푸시까지 이어지는 엔드투엔드 처리 선호가 강함
- UI는 둥글고 가벼운 느낌보다 각지고 밀도 높은 "메이저 제품" 톤을 선호함
- 배포 문제 대응 시 추상 설명보다 로그 기반 원인 확정 + 바로 실행 가능한 체크리스트를 선호함
- 코딩 비경험 사용자 기준으로 안내 필요:
  - "무엇을 어디서 눌러야 하는지"를 화면 경로 단위로 구체적으로 설명
  - 확인 요청 시 기대 결과(정상 예시/문제 예시)를 함께 제공
  - 가능한 경우 복붙 가능한 명령/URL/쿼리를 우선 제공
  - 기술 용어를 쓸 때는 한 줄 설명을 같이 붙여 오해를 줄임

## AI 작업 방식 가이드

- 구현 디테일은 자율 결정하되, 위 목적과 품질 목표를 우선한다.
- 기능 추가 전, 기존 사용자 흐름을 먼저 보존한다.
- 불안정한 부분은 임시 대응보다 재발 방지 구조로 해결한다.
- 완료 보고는 “무엇이 좋아졌는지”를 사용자 관점으로 짧게 전달한다.

## 재발 방지 소통 규칙 (2026-03-06)

- 질문 의도 확인 우선:
  - 사용자가 말한 핵심 요구(예: "기술 말고 소통")를 먼저 1문장으로 재확인하고 작업 시작
- 반영 안 됨 이슈 응답 순서 고정:
  1. 커밋 여부
  2. 푸시 여부
  3. 배포 반영 여부
  4. 캐시/새로고침
- 중간 보고 형식 고정:
  - 현재 확인한 사실 1개 + 다음 행동 1개만 짧게 전달
- 결과 보고 형식 고정:
  - 원인 / 조치 / 사용자가 바로 확인할 방법 순서로 답변
- 사용자가 "나중에"라고 한 항목은 즉시 보류 처리:
  - 해당 이슈는 더 파고들지 않고 현재 요청만 집중 처리
- 사용자가 제공한 로그/문구를 우선 근거로 사용:
  - 추측보다 로그의 결정적 문장을 먼저 인용해 설명

## 이력 관리 (필수)

- 프로젝트 시작 시 루트에 `HISTORY.md` 파일을 반드시 생성한다.
- 작업 중간/완료 시점마다 변경 이력을 누적 기록한다.
- 기록은 최소 아래를 포함한다.
  - 날짜
  - 변경 목적(무엇을 왜 바꿨는지)
  - 사용자 체감 변화(무엇이 좋아졌는지)
  - 남은 리스크/다음 작업
- 이력 기록 없이 기능만 반영하는 작업은 금지한다.

## 추가 권장 항목

- `KNOWN_ISSUES.md`: 현재 재현 가능한 버그와 임시 우회책을 분리 기록
- `DECISIONS.md`: 큰 구조/정책 결정과 선택 이유를 짧게 기록
- 릴리즈 전 체크리스트: 로그인, 저장, 새로고침, 검색, 필터, 복원, AI 분석 기본 시나리오 점검
