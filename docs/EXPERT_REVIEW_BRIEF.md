# Expert Review Brief

## Purpose

이 문서는 LinkPocket 개선을 위해 외부 전문가에게 프로젝트를 설명하고, 분석/피드백을 받기 위한 임시 브리프다.

목표는 아래 두 가지다.

1. 현재 제품의 강점/약점/위험요소를 빠르게 진단받기
2. 우선순위가 높은 개선안을 기능, 디자인, UX, 기술, 운영 관점에서 도출하기

## Project Summary

- 제품명: `LinkPocket`
- 제품 성격: AI 보강 기능이 포함된 개인 링크/기사 저장 서비스
- 핵심 사용 흐름:
  - 링크 저장
  - 저장된 링크를 읽기 상태/컬렉션/태그로 관리
  - AI가 제목/요약/카테고리/키워드 보강
  - import/export로 데이터 이동
- 현재 기술 스택:
  - Frontend: React + Vite + TypeScript
  - Backend/API: Cloudflare Workers
  - Auth/DB: Supabase

## Current Product Intent

이 프로젝트가 지향하는 가치는 단순 북마크 저장이 아니라 아래에 가깝다.

- 링크를 “쌓는 것”이 아니라 “다시 읽고 분류하고 활용하는 것”
- 저장 직후 AI로 요약/정리해, 나중에 다시 볼 비용을 줄이는 것
- 읽기 상태, 컬렉션, 태그, 카테고리를 통해 개인 지식 저장소처럼 쓰게 하는 것

## Current State

현재는 핵심 기능은 대체로 존재하지만, 유료 제품 수준의 완성도보다는 “구조 정리 중인 내부 베타”에 가깝다.

상태 요약:

- 링크 저장/목록/상세 편집 가능
- 읽기 상태, 즐겨찾기, 태그, 컬렉션 관리 가능
- AI 분석/요약/카테고리 보강 가능
- 파일 import/export 가능
- 대형 `App.tsx`를 단계적으로 분리하는 리팩터링 진행 중

## What Needs Expert Analysis

전문가는 아래 항목을 중심으로 봐주면 된다.

### 1. Product Positioning

질문:

- 이 제품은 사용자에게 한 문장으로 어떻게 설명되어야 하는가?
- “그냥 북마크 앱”과 구분되는 핵심 가치가 충분히 선명한가?
- 유료화가 가능하려면 어떤 가치가 더 명확해져야 하는가?

봐야 할 포인트:

- 문제 정의의 선명도
- 타겟 사용자 정의
- 차별점
- 과금 가능성

### 2. Core Feature Quality

질문:

- 현재 기능 구성은 실제 사용 흐름에 맞는가?
- 가장 먼저 강화해야 할 핵심 기능은 무엇인가?
- 제거하거나 단순화해야 할 기능은 무엇인가?

봐야 할 포인트:

- 링크 저장 UX
- 상세 편집 UX
- 태그/컬렉션/카테고리 구조의 이해 가능성
- import/export의 신뢰성
- AI 요약/분류의 체감 가치

### 3. UX / Information Architecture

질문:

- 화면 구조가 처음 보는 사용자에게 직관적인가?
- 정보량이 많은데도 탐색이 쉬운가?
- 어디서 가장 피로감이 생길 가능성이 높은가?

봐야 할 포인트:

- sidebar / topbar / list / detail 구조
- 검색/필터/정렬 흐름
- 다중선택과 일반선택의 충돌 여부
- 설정 화면의 정보 과밀도
- 모바일 사용성

### 4. Visual / Brand Design

질문:

- 시각적으로 돈 받고 제공할 서비스처럼 보이는가?
- 브랜드 인상이 약한가, 과한가, 애매한가?
- 신뢰감/생산성/고급감 중 어떤 방향이 더 맞는가?

봐야 할 포인트:

- 색/타이포/간격의 일관성
- 리스트와 상세 패널의 시각적 우선순위
- 빈 상태, 로딩 상태, 에러 상태 표현
- 브랜드 개성
- 아이콘/로고/시각 언어

### 5. AI Feature Evaluation

질문:

- AI 기능이 진짜 가치를 만들고 있는가, 아니면 부가 기능 수준인가?
- 사용자가 AI 결과를 신뢰할 수 있게 하려면 무엇이 필요한가?
- AI 비용 대비 효용이 충분한 구조인가?

봐야 할 포인트:

- 요약 품질
- 카테고리 정확도
- 키워드/태그 활용성
- AI 실행 시점과 피드백 UX
- 실패 시 복구/재시도 UX

### 6. Reliability / Trust

질문:

- 사용자가 데이터를 맡겨도 된다고 느낄 수준인가?
- 어떤 지점에서 신뢰가 가장 쉽게 깨지는가?

봐야 할 포인트:

- autosave 신뢰성
- 삭제/복원 흐름
- import/export 안전성
- 에러 메시지의 명확성
- 로그인/세션 만료 대응

### 7. Technical / Delivery Risk

질문:

- 현재 구조에서 개발 속도를 가장 크게 막는 지점은 무엇인가?
- 지금 리팩터링 방향이 맞는가?
- 다음 4주 동안 기술적으로 꼭 해결해야 할 것 3개는 무엇인가?

봐야 할 포인트:

- 대형 컴포넌트 구조
- query/action/view 분리 방향
- 테스트 부족
- 배포/환경변수/운영 리스크
- 장애 복구 가능성

## Suggested Feedback Format

전문가는 아래 형식으로 답하면 된다.

### A. One-line diagnosis

예:

- “방향은 좋지만 아직 유료 제품보다는 내부 베타에 가깝다.”
- “AI 가치 제안은 흥미롭지만, 현재는 정보 구조와 신뢰성이 먼저다.”

### B. Top strengths

- 현재 가장 좋은 점 3개

### C. Top weaknesses

- 현재 가장 치명적인 약점 3개

### D. Priority recommendations

- 지금 당장 해야 할 일 3개
- 1개월 내 해야 할 일 3개
- 하지 말아야 할 일 3개

### E. Paid-product readiness

- 지금 돈 받고 팔 수 있는지
- 아니라면 무엇이 먼저 충족되어야 하는지

## Specific Questions To Ask

아래 질문을 그대로 전달해도 된다.

1. 이 제품의 핵심 가치는 한 문장으로 무엇이어야 하나?
2. 현재 구조에서 사용자가 가장 먼저 헷갈릴 화면은 어디인가?
3. AI 기능은 차별점으로 충분한가, 아니면 보조 기능인가?
4. 지금 유료화가 이르다면, 어떤 기준이 충족돼야 하는가?
5. 디자인적으로 가장 먼저 손봐야 할 부분은 어디인가?
6. 기능적으로 가장 먼저 강화해야 할 부분은 어디인가?
7. 신뢰성 측면에서 가장 위험한 구간은 어디인가?
8. 기술적으로 지금 리팩터링 방향이 타당한가?
9. 당장 줄여야 할 복잡도는 어디인가?
10. 4주 안에 제품 가치를 가장 많이 올릴 개선안은 무엇인가?

## Reviewer Notes

전문가에게 아래 전제를 같이 알려주면 좋다.

- 현재는 완성품이 아니라 구조 정리와 제품 방향 검증이 동시에 진행 중인 단계다.
- 기능 추가보다 “유료로 쓸 만한 제품인지”를 판단할 수 있는 진단이 더 중요하다.
- 예쁜 평가보다, 냉정한 우선순위와 버릴 것/남길 것을 구분해주는 피드백이 필요하다.
